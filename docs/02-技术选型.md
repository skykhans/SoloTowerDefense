# 技术选型说明

## 当前原型技术

当前仓库使用的是轻量 Web 原型方案：

- HTML：页面结构
- CSS：界面样式
- JavaScript：游戏逻辑
- Canvas 2D：战斗画面绘制
- Python `http.server`：本地静态服务器

## 为什么当前原型适合前期验证

- 依赖少，上手快
- 适合快速试数值和玩法
- 容易直接修改逻辑并观察反馈
- 成本低，适合快速试错

## 当前原型的限制

- DOM UI 和 Canvas 绘制耦合在同一份脚本中
- 不适合作为正式微信小游戏工程直接上线
- 难以扩展资源管理、复杂动画和平台接口
- 无法直接复用网页 UI 到微信小游戏运行时

## 微信小游戏推荐技术方案

推荐路线：

- 引擎：Cocos Creator 3.x
- 语言：TypeScript
- 平台：微信小游戏
- UI：Cocos 节点与 UI 系统
- 数据：TS 配置表或 JSON 配置
- 存储：前期无后端，优先本地存储，后续再接云开发或自建后端
- 平台封装：`platform/` 目录统一收口微信与 Web 差异

## 选择 Cocos Creator 的原因

- 国内微信小游戏生态成熟
- 2D 游戏支持稳定，适合塔防
- 资源管理、动画、场景、打包和平台适配流程完整
- 相比 Unity 更轻量，包体和性能更适合中小型微信小游戏

## 不建议直接延用的部分

- `index.html`
- `styles.css`
- 基于浏览器 DOM 的按钮和布局
- 浏览器事件模型

## 可复用的部分

- 炮塔与敌人属性设计
- 波次规则
- 伤害、奖励、生命值与金币模型
- 敌人路径与塔的选敌逻辑

## 无后端阶段的数据存储策略

前期不接传统后端，数据优先保存在本地：

- 最佳波次
- 已解锁关卡
- 当前选择关卡
- 局外货币
- 已解锁炮塔
- 新手引导状态
- 音频设置

在微信小游戏环境下优先使用 `wx` 本地存储；在非微信环境下回退到本地浏览器存储。

## 当前平台适配策略

当前已经新增：

- `assets/scripts/platform/`

作用：

- 统一识别运行环境
- 统一封装本地存储
- 避免业务层直接散落访问 `wx`
- 为后续分享、震动、系统信息、登录等平台能力预留入口

当前已落第一批能力入口：

- `SystemInfoService`
- `VibrationService`
- `ShareService`
